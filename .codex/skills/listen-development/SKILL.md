# Listen Development

每次接手都当作新项目。

先看事实。再做判断。最后行动。

事实来自当前 `spec.md`、`AGENTS.md`、`plan.md`、`README.md`、`miniprogram/`、`pages/`、`cloudfunctions/`、`tests/`、`package.json`、`tsconfig.json`、`.codex/skills/`、`skills/`、git 状态、命令结果和工具反馈。

用户偏好、用户一句话、旧计划、当前实现、测试绿灯、历史设计，都不是绝对真理。它们只是证据。结论必须来自证据之间的一致性。

## 铁律

- 先 research，再计划，再实现。
- 没有完成全局核心语义调查，不动局部代码。
- 用户输入是线索，仓库事实决定行动。
- 抓根本逻辑，不追表面症状。
- 质疑用户判断，质疑当前代码，质疑旧设计，质疑测试结果，质疑自己的第一反应。
- 悬置判断，直到证据收束。
- 禁止把用户话术直接写成硬编码规则、提示词、正则、特殊分支或局部补丁。
- 只有产品行为、架构边界、代码事实和验证方式共同支持时，判断才进入实现。
- 改代码前优先参考微信小程序、CloudBase、TypeScript 和当前仓库已有模式；禁止没有 research 就从零生造。
- 不交“先搭骨架后优化”的半成品。架构、类型、测试、文档和验证必须作为同一次交付的一部分。

## 当前产品主干

只写当前事实主干。

当前产品是微信小程序 + CloudBase 的英语听写工具：

- 系统库和用户库是材料来源边界。
- 通用听力属于系统库。
- 用户可以新增分类，分类由 `SourceLibrary` 承担。
- 文本块是材料单位。
- 导入内容按完整材料保存。
- 材料是播放和音频绑定的基本单位。
- 一份材料可以绑定一份或多份音频。
- 音频来源包括用户上传和系统生成。
- 上传音频支持 `mp3`、`m4a`、`wav`。
- TTS 是缺音频时的补全工具，不是唯一音频来源。
- 本地接口和云端接口分离，不自动同步。
- 管理入口必须校验管理员 `openid`。
- `local-assets/<folder-id>/` 的文件夹名是样例材料唯一标识和默认标题来源，材料文件使用 `text.txt`、`audio.mp3|m4a|wav`、`image.svg|png|jpg|jpeg`。

当前不做：

- 听写输入。
- 音频导出。
- 口语评分。
- 发音纠错。
- 语法教学。
- 整篇音频自动对齐。
- 音频自动切句。
- 图片 OCR。

当前没有的能力，不进入源码、测试、文档、计划、README 或 UI 文案。

## TypeScript 铁律

本项目要求强可维护性和扩展性。工程代码默认使用 TypeScript。

- 领域模型必须有显式类型。
- 小程序端和云函数之间的请求/响应必须有共享契约。
- 状态枚举必须使用联合类型或常量对象约束。
- TTS provider 必须是可替换接口，不允许页面直接调用具体供应商。
- 页面文件只负责页面状态和用户交互，不承载数据访问、播放器规则、上传规则、统计规则或 provider 逻辑。
- 纯业务规则放在可测试模块中。
- 云函数必须通过共享 response helper 返回稳定结构。
- 禁止 `any` 逃避设计；如必须使用，必须说明原因并限制在接线边界。

## 全局语义调查

调查不是扫一眼。

从整体到部分检查：

- 产品事实：`spec.md` 说了什么，不做什么。
- 计划事实：`plan.md` 当前目标、失败测试、任务和收口是什么。
- 小程序入口：页面如何进入用户路径。
- 页面边界：页面是否只做交互和呈现。
- 服务边界：小程序端 service 是否负责 API、缓存、上传、播放和统计接线。
- 类型边界：domain/api 类型是否覆盖所有跨模块数据。
- 云函数边界：云函数是否只做鉴权、数据读写、provider 调用和稳定返回。
- TTS 边界：未配置密钥时是否明确失败，不伪装成功。
- 数据边界：云端为准，本地缓存是否只做恢复和加速。
- 测试边界：测试是否保护真实产品行为，而不是保护口号。
- 文档边界：README/spec/AGENTS 是否讲同一个当前事实。

审查过程按核心链路推进：

打开小程序 -> 选择本地或云端入口 -> 拉取资料库 -> 导入完整材料 -> 上传或生成材料音频 -> 在材料页播放或查看原文 -> 管理删除 -> 文档和测试验证。

## 文件职责

单一职责看变化原因，不看行数。

超过 300 行必须触发职责审查，但不是自动拆分理由。

推荐边界：

- `miniprogram/types/`：领域类型和 API 契约。
- `miniprogram/lib/`：纯函数规则。
- `miniprogram/services/`：小程序端外部接线和业务动作。
- `pages/*`：页面状态、事件处理和渲染数据。
- `cloudfunctions/_shared/`：云函数共享类型、响应、鉴权、provider 接口。
- `cloudfunctions/<name>/`：单个云函数入口。
- `tests/`：真实行为测试和机器检查。

状态管理、规则计算、数据读写、渲染展示、外部接线、错误恢复、业务判断混在一起时，必须拆清边界。

不为了拆而拆。不为了省事硬塞。

## 交付标准

接到明确问题后，把 research、设计、实现、测试、文档同步和验证收成一个完整交付。

不交半成品。

把“顶尖标准”翻成可验收的终局，不写成“继续优化”。

把任务定成生产级封顶验收，不写成后续优化或逐步改进。

不能一次闭环时，说明客观阻塞、已完成事实和剩余风险。

大改完成前运行项目完整验证命令。当前项目的完整验证命令必须在 `package.json` 中定义为：

```powershell
npm.cmd run verify
```

commit / push 只在项目所有者明确要求时执行。

## Caveman

短。准。硬。

少废话，不少判断。

少解释，不少证据。

少抽象，不少边界。

说不清，先别改。
