# 每日英语听写 Agent 工作规约

本文件是本仓库的最高工作约束。产品事实看 `spec.md`，当前任务执行状态看 `plan.md`。

## 基础约束

- 始终使用简体中文和项目 owner 交流。
- 阅读本仓库文本文件时按 UTF-8 读取；PowerShell 中文显示乱码时，不要误判文件损坏。
- 在 PowerShell 中运行 Node/npm 命令时使用 `npm.cmd`、`npx.cmd`，不要直接用 `npm`、`npx`。
- 不回滚用户已有改动，除非 owner 明确要求。
- 遇到脏工作区，只处理当前任务相关文件。
- 不使用阶段性版本命名组织产品事实、计划或回复。直接使用标准功能名、模块名和任务名。
- 禁止跨文件机械替换、批量正则替换或大小写不受控的命名迁移。命名调整必须逐文件语义修改，并通过类型检查、边界扫描和必要搜索验证。

## 根文档边界

当前项目的根文档职责如下：

- `AGENTS.md`：agent 工作规则和仓库协作纪律。
- `plan.md`：当前任务的单文件规格驱动执行合同。
- `spec.md`：当前产品事实、业务边界、验收标准和开源协议。
- `plan.example.md`：后续任务计划模板。
- `README.md`：面向新读者的项目入口。
- `LICENSE`：开源协议权威文本。
- `SECURITY.md`：安全报告和敏感数据处理规则。
- `CONTRIBUTING.md`：贡献流程和文档驱动规则。
- `.gitignore`：不进入版本控制的文件边界。
- `.codex/skills/listen-development/SKILL.md`：本项目开发、架构、TypeScript、CloudBase、音频绑定和生产级交付规则。
- `.codex/skills/listen-ui/SKILL.md`：小程序 UI、WXML/WXSS、Vant 组件、页面布局和按钮语义规则。
- `.codex/skills/plan/SKILL.md`：`plan.md` 写作和执行规则。

除非 owner 明确要求，否则不创建额外 skill 体系或无关工程目录。

## Skills

项目级 skill 放在 `.codex/skills/`。

命中 skill 描述时，必须先读取对应 `SKILL.md`，再行动。

- 修改微信小程序、CloudBase 云函数、TypeScript 类型、数据模型、音频绑定、TTS provider、缓存同步、管理入口、spec、tests、README、AGENTS.md、skills 或运行配置时，先读 `.codex/skills/listen-development/SKILL.md`。
- 修改 WXML、WXSS、Vant 组件、页面信息架构、入口卡片、材料列表、导入表单、详情页、按钮、空状态或视觉样式时，先读 `.codex/skills/listen-ui/SKILL.md`。
- 中大型开发、跨模块重构、生产级验收、恢复演练、架构硬化、评测闭环，或任何不能靠一次小补丁完成的任务，先读 `.codex/skills/plan/SKILL.md`，并维护根目录 `plan.md`。
- 如果 skill 和 owner 当前明确需求冲突，以 owner 当前需求为准，但必须先更新 `plan.md` 记录冲突和新事实。

## 事实纪律

- 事实高于一切。所有判断、计划、文档修改和回复必须基于当前仓库文件、命令输出、用户明确提供的信息或可复现结果。
- 必须区分：已确认事实、用户提供事实、未验证推测、计划事项、已完成并验证事项。
- 事实不清楚时先排查；无法排查时明确说明不确定。
- 不编造功能、数据、接口、页面行为、测试结果、服务能力或业务规则。
- 不把未实现能力写成当前产品事实。
- 不把历史讨论、外部项目风格或用户建议自动写成当前产品事实；它们只能作为证据来源。
- 文档只写当前事实和当前边界，不写假兼容、旧状态、旧数据或没有落点的未来设想。

## Plan 规则

`plan.md` 是单文件规格驱动执行合同。中大型任务、跨文档修改、产品边界调整、工程实现、验证闭环都必须先维护 `plan.md`。

执行规则：

- 写完计划，再动手。
- checklist 必须随进度更新。
- 新事实推翻计划，先改计划，再继续。
- 不用口头解释替代计划更新。
- 不偷偷偏离计划。
- 不把旧能力、旧状态、旧数据写成当前产品事实。
- 不为了套模板制造多余章节；无关章节写明“不适用”和理由，或删掉。
- 不为了简短牺牲边界、失败测试和验证。
- 提出一个方向后，要把该方向一次性做成完整闭环：research、设计、实现、测试、文档同步、验证和收口都完成。
- 不把方向拆成反复让用户推动的小步。除非遇到客观阻塞，否则同一方向做到可交付、可验证、可长期保留。
- 计划的任务边界必须足够完整，避免今天补一点、明天再补同一主线的尾巴。

`plan.md` 必须至少覆盖：

- 需求文档。
- 当前事实。
- 失败测试。
- 目标。
- 不做范围。
- 设计。
- 实施任务。
- 验证计划。
- 收口。

## Spec 规则

`spec.md` 是产品事实主干。

写入 `spec.md` 时必须遵守：

- 只写当前产品定位、用户路径、功能边界、数据边界、验收标准和明确决策。
- 不写实现演进过程。
- 不写尚未选择的技术栈为既定方案。
- 不写尚未存在的页面、接口、数据库、云函数或第三方服务为当前能力。
- 产品功能必须对应真实用户路径。
- 不把语法教学、口语评分、发音纠错、社交、课程商城或复杂考试系统写成当前主线。
- 如果 owner 改变产品方向，先更新 `plan.md`，再同步 `spec.md`。

## 产品纪律

当前产品名是“每日英语听写”。

当前产品主线是：把英文文本和音频绑定成可播放、可盲听、可循环、可变速、可复听、可记录的英语听写材料。

默认判断：

- 听写是主线。
- 系统库和用户库是材料来源边界。
- 通用听力属于系统库。
- 材料是播放和音频绑定的基本单位。
- 用户上传音频和系统生成音频都是合法音频来源。
- TTS 只是缺音频时的补全工具，不是唯一音频来源。
- 阅读是材料查看能力，不是独立训练主线。
- 跟读是用户自然行为，不做评分主线。
- 语法教学不进入当前产品。
- 词义查询只在已有词库命中时呈现，不强行联网解释。
- 图片是文本块附件，不参与文本转语音和听写评分。
- 不做整篇音频自动对齐或音频自动切句。

## UI 纪律

- 原生 `button` 和 `van-button` 只用于真实命令，不当作卡片、列表项、网格入口或整行布局容器。
- 可点击入口和列表项使用 `view bindtap`，并用业务 class 命名。
- 修改页面结构前先看对应 `.wxml`、`.wxss`、`.ts`、`.json` 和上下游页面，不能只靠改颜色处理 UI 问题。
- 横向布局必须明确 `min-width: 0`、`flex`、固定宽度和换行策略，避免文字被挤压、竖排或溢出。
- 禁止把上一次实现的坏样式靠局部覆盖保留下来；当前产品不需要的 UI 结构直接删除。

## 验证规则

- 文档任务必须检查目标文件是否存在。
- 文档任务必须检查是否误写未实现能力为当前事实。
- 文档任务必须检查是否偏离当前根文档职责和用户指定范围。
- 代码任务出现后，收尾前必须运行项目定义的类型检查、测试、构建或等价验证；无法运行时说明原因和风险。
- 不能只凭解释声称完成，必须说明验证结果。
